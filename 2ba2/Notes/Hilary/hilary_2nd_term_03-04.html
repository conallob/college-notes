1. Heapsort

Like Quicksort Heapsort is an O(n log n) sorting algorithm. It has the advantage over Quicksort in that its worse case performance is still O(n log n). Also Heapsort can be written non-recursively without the overhead of an explicit stack-like structure. Heapsort is strictly in-place sort which may not be the case with Quicksort due to the recursive calls or the use of an implicit or explicit stack.

Sorting via Heapsort is based on the idea of viewing an array as a 'binary tree' with index/node 1 as the root and the 'children' of node i being node 2i and 2(i +1). The array is fist converted to a 'heap', then items in the array are continually swapped and 'heapified' until the whole array is sorted.

Definition:      Heap

            It is best to view a Heap in binary tree terms even though it is implemented on an array.
A binary tree is a Heap iff

  • It is complete (i.e. Balanced) in the sense that if n = #nodes then height of tree £ é log nù
    Note: Height of tree = max level of all nodes in the tree (level of root = 1)
    e.g.       n = 10, Height of Heap (tree)     £ é log 10ù = 4.
    In a Heap the lowest or 'leaf' level is filled from left to right.
  • It has the ‘Heap Property’, i.e. for each node k, value at k ³ value of each child (if any) of of k.

When an array has the heap-property, the item A@1 is the largest item in the array. If we replace A@1 with A@n we destroy the heap property at node 1. But the subtrees at A@2 and A@3, i.e. the subtrees of the children of node 1 are still heaps. To restore the heap property we must ‘sift down’ the value of the node at 1 down to its proper position in the array. This is what the procedure Heapify (the key procedure in Heapsort), in effect, does.
            We can describe Heapsort in template form as follows;
Assume we have array attribute A of items of type G with A.lower = 1 and A.upper = n.

Heapsort is
   local
           ...
   do
           <Build or convert A into a Heap>
           from
                   i := n 
                   until
                   i = 1
           loop
                   <Exchange A@1 and A@i> 
                   i := i-1
                   Heapify(1, i)
           end
   end-Heapsort

2. Priority Queue

The class has the following interface or specification

class PRIORITY_QUEUE [G -> COMPARABLE]

creation make

feature

count_PQ : INTEGER -- #items in the Priority_Queue
make -- create a Priority_Queue of default size
empty : BOOLEAN       -- Is the Priority_Queue empty
-- in effect, is count_PQ = 0
add (x : G)   -- Insert x into the Priority_Queue
item : G -- Get ‘largest’ item from the Priority_Queue
remove        -- Remove ‘largest’ item from Priority_Queue

end -- PRIORITY_QUEUE

               A Priority_Queue is in effect a suite or bag of items such that we can get and remove the largest item. A suite or bag of items is a 'set' where repeated items are allowed.

Heap Implementation

In the hidden part of the class assume we have an array

heap : ARRAY[G]

Also let

default_size : INTEGER is 16.

We will extend the array when necessary.

The array, heap, will be maintained as a heap. This will mean that the routines remove and add will be O(log n).

Implementing a Priority Queue by a (linked) List

               Implementing a Priority Queue by an array with the ‘Heap Property’ is very efficient except that the array may have to be resized.

               Implementation by a linked list is possible. By maintaining the list in descending order, the largest item will be at the front of the list. In this case removing an item is very efficient but adding an item is more expensive as each insertion of an item into a ordered list will perform at O(n).

               In implementing Heapsort on linked list, it will take more effort, O(n^2), to built the queue, while the sorting phase will be O(n).

3. List Set

An Abstract Data Type (ADT) for a set is implemented via linked lists, without a header node. The short form of the ADT LIST_SET is

class interface
        LIST_SET [G]
feature 
 
        add (x: G) -- Add x if not already there
 
        has (x: G): BOOLEAN
 
        remove (x: G) -- remove only occurrence of x, if any.
 
        count: INTEGER
 
        empty: BOOLEAN
 
-- Traversal routines
 
        first: G  -- The item at first_node
 
        forth -- move cursor forward
 
        item: G -- item at cursor
 
        finish -- set cursor to last node
 
        last: G -- return last item in list
 
        off: BOOLEAN -- Is cursor beyond end
 
        start -- set cursor back to start
 
-- inherited routines
 
        copy (other: like Current)
 
        is_equal (other: like Current): BOOLEAN
        
end -- class LIST_SET

 

Difficulties with the cursor:

Introducing a cursor may cause side-effects in functions. For example, if the cursor is at at a particular item in the middle of the list and we have a function that returns the last item in the list we have to be careful not to move the cursor.

A more difficult problem is when we remove the node/item at the cursor. When we remove an item do we leave the cursor at the left or right of the removed item. If the list has one item then there is no left or right item when the item is removed.

4. Inheritance in Eiffel 

Eiffel supports multiple inheritance, i.e. a class C can inherit from more than one class. Inheritance in used in two major ways:

  • Specialization
    A class C is a specialization of a class P, if C has all the properties (attributes & features) of P plus maybe a few more defined itself.
    The extra properties would distinguish an object from class C from one from class P.
  • Generalization
    We may have already, for example, the classes Salesperson, Secretary etc, and decide that these are specializations of a class Employee; we generalise from the particular to the general. Common features in the particular class can be moved up to the general class.
    In particular, the more general class may have deferred features and so it will be a deferred class.
    The most general class in Eiffel is called GENERAL    .

The Class Hierarchy in Eiffel

In Eiffel all classes are classified into an hierarchy, analogous to the "Linnean classification"of livings things. Every class inherits from the class ANY. We say, every class ‘conforms’ to ANY.
The class NONE is a fictional or virtual class, there is no text in the class, which is a descendant of every class, i.e. it inherits from every class or conforms to every class. In the class GENERAL there is a declaration
               frozen void : NONE
but this is viewed as a dummy declaration.
If NONE inherits from every other class then it has the properties of every single class (even classes that have not been written yet?). In particular,we can regard void as being in every class.

Deferred Classes

These universal classes are so general that they have no implementation and so are "deferred" classes. The implementations are deferred until created by later conforming classes, e.g. in the class LIST_SET, it is possible to implement the 'deferred' routines e.g. ‘copy’ and ‘is_equal’.

Deferred routines can be written with just the precondition and postcondition assertions, so that the implementation can be written to satisfy these assertions. In this way, the specification of a routine can be written before its implementation.

5. ADT/Class LIST_BAG

The class LIST_SET did not allow repeated items, each item appeared once in the list. We consider a related class which allows repeated items and these items are implicitly kept in a ‘Last-in First-out’ order. As an example of using LIST_BAG, we give a class for sorting (via Quicksort) the items in the list.

The class LIST_BAG is a descendent of the class LIST_SET and so all the routines in LIST_SET are available in LIST_BAG. In order that we can implement a new routine, join, more efficiently, we change the ‘export’ rules of the inherited attributes, first_node and cursor, so that they are available to the class LIST_BAG itself.

We also have to redefine the routine, add, as previously this routine checked if the list had the item already. The new version of the routine just puts the new item at the front of the list.

The routine, remove, is as before but in the case of LIST_BAG, the routine removes only the first occurrence of an item, if it is in the list.

 6. Binay Search Tree (BST)

In a Binary Search Tree (BST), all the values in the Left subtree are (strictly) less than the value at the root which is (strictly) less than the values in the right subtree.

In adding/inserting an item into a tree we insert it into the left subtree if it is less than the root value and into the right subtree if it is greater.
If the value of the item is equal to the root value, then no insertion takes place. The binary tree contains a ‘set’ of items, there are no repeated items.

The pattern for recursive delete is similar to that of recursive update/insert except for the special case of deleting a 'root' node. If x is the value of a root node that has a non-void left subtree then to delete x we replace the value x of the root with the value of its predecessor in the tree, and delete the predecessor node. In a BST, the predecessor value of a node with a left subtree is found in the Right_Most node of the left subtree.

Tree Traversal

In a Binary Tree we can traverse the tree in either of 3 main ways; Preorder, Inorder or Postorder. Let Left and Right be the left and right subtrees.

  • Preorder:
    Root first then Preorder Left then Preorder Right
  • Inorder:
    Inorder Left then the Root then Inorder Right.
  • Postorder:
    Postorder Left then Postorder Right then the Root

If the tree is a BST, then Inorder traversal traverses the tree in ascending order.

7. Knight's Journey

Problem:  (See Wirth "Algorithms & Data Structures" Ch 3.4)
Given an NxN "chessboard" (N^2 squares), determine, if possible, all the moves starting from (x0,y0) such that the knight visits all squares on the board. Can a knight starting from (x0, y0) visit all the other (N^2-1) squares on board exactly once, i.e. the knight visits all the squares and visits each square just once. Can knight go on a journey visiting all the squares. This may not be possible, and the program should indicate so.

If the knights journey ends within a knights move of the start then we consider we have a tour or a closed journey.

A pseudo-code solution can be given as:

Try(k, x, y)  -- attempt move k from (x,y)
           ...
   do
           if "Board Full" then 
                   "Finished"
           else
                   from
                          "start move"
                   until
                          "No more Moves" or "success"
                   loop
                          "Select next Move"
                          If "acceptable" then
                                  "record move"
                                  Try(k+1, "New (x,y)")
                                  If not "success’ then
                                         "reset  New(x,y)"
                                  end
                          end
                   end -- loop
           end
   end -- Try

 

Related Knight Journey Problems

  • The Circular Journey or the Knights Tour. (tour = travel round;         journey = go to a place)
    Is there a tour starting from say (1,1) that ends within a knights move of (1,1). On a 8x8 board the 65th move could lead back to the start.
    Why is there no (circular) tour for a NxN board, when N is odd.?
  • Find all solutions to the knights journey problem.
    The above routine, Try, finds the first solution. Can this routine be adapted to find all knights journeys/tours around the board?
  • Delete Opposite corners of the board.
                Is there a knights journey when opposite corner squares are removed.

8. Generating Permutations ("Rook's Problem") & Derangements

Finding all solutions to the 'Rooks Problem' of placing N non-attacking rooks on a chess board is the same as generating all the permutations of 1..N.
A solution to the rooks problem is an injective function, R, onto the set {1..N} where the domain of R is also {1..N}. Since #Dom(R) = #Ran(R), R is a Bijective function, tf. R is permutation on {1..N}
To find all solutions of the Rooks Problem we need a program that will generate all permutations of {1..N}

Permutation

A permutation on 1..n is a bijective function on 1..n. Each argument has just one value, and each element in 1..n is the image of some value in 1..n.

Related Problems

  • Find all permutations without using backtracking (but still using recursion).
  • Find all combinations of 1..n.
  • A Derangement of 1..n is a permutation p s.t. p(i) ¹ i
    Calculate the number of derangements of 1..n
    Generate all Derangements of 1..n..

Counting derangements

Derangement

A Derangement of 1..n is a permutation, p, s.t. p(i) ¹ i, i.e. p has no 1-cycle.

A permutation is a derangement if no element maps onto itself.

It can be shown that the number of derangements of 1..n, D(n), satisfies each of the following:

  • D(1) = 0,
    D(2) = 1
    D(n) = (n-1)(D(n-2) + D(n-1)),            for n>2
  • D(1) = 0
    D(n) = n D(n-1) + (-1)^n, for n>1
  • D(n)     = round (n!/e) -- the nearest integer, n>1

e.g. D(4) = 9; D(5) = 44

9. N-Queens Problem

Task:
Place N queens on a N x N chessboard so that no queen checks another.
Two Queens check each other if,

  • They are in the same row,
  • They are in the same column,
  • They are on the same diagonal.

 As in Rooks problem we use an array, q : ARRAY[INTEGER], to represent a solution, where q.item(i) denotes the column position of the i-th queen on the i-th row, i.e. j= q.item(i) iff a queen is on col j in row j. This guarantees one queen per row.

On an 8x8 board, one solution is 3 5 2 8 1 7 4 6.

10. Unique Queens Solutions

Out of the 92 solutions to the 8-queens problem, only 12 are essentially different, in the sense that none of these 12 solutions are symmetries of each other. There are 8 symmetries of a square, 4 rotations (clockwise: 0° , 90° , 180° , 270° ) and 4 reflections (horizontal, vertical, up-diagonal, down-diagonal). These 8 symmetries can be generated by 'rotate' (90° rotation-clockwise) and 'mirror' (horizontal reflection).

The 12 unique solutions (in sequence form) are:

[2 4 6 8 3 1 7 5]

11. Generate Subsets

Given a set of N elements, we want a procedure that will generate all the 2^N subsets.

For representing sets as a boolean array assume we have an attribute

s : ARRAY[BOOLEAN] -- Integer Set: s.item(i) iff i e s

A procedure for generating subsets can be given as:

        Gen_Subsets(i,N : INTEGER) is
               do
                       if i > N then
                               Print_Set
                       else
                               s.put(False,i) -- exclude i
                               Gen_Subsets(i+1,N)
                               s.put(True,i)  -- include i
                               Gen_Subsets(i+1,N)
                       end
               end -- Gen_Subsets

 

Combinations (again)

To generate all the combinations of k numbers from N numbers, generate all the subsets of {1..N} and output only those of size k.
